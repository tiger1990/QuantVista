# 04 — API Contracts

> Contract-first. OpenAPI (generated by FastAPI) is the source of truth; the frontend consumes a generated
> typed client. REST/JSON, versioned under `/api/v1`.

---

## 1. Conventions

- **Base:** `https://api.quantvista.app/api/v1`
- **Auth:** `Authorization: Bearer <access_jwt>`. Access token short-lived (~15 min); refresh token rotated
  (httpOnly cookie for web; refresh endpoint for API clients). Quant-tier API keys for programmatic access.
- **Tenant context:** derived from the token (principal → active tenant). No tenant id in the URL.
- **Content type:** `application/json`. Dates ISO-8601 (UTC); the UI localizes to IST.
- **Idempotency:** mutating endpoints accept `Idempotency-Key` header; replays return the original result.
- **Pagination:** cursor-based — `?limit=50&cursor=<opaque>`; response carries `meta.next_cursor`.
- **Filtering/sorting:** `?filter[roe][gte]=15&sort=-momentum_score`.
- **Rate limiting:** per-tenant + per-plan; `429` with `Retry-After`; headers `X-RateLimit-*`.

### Standard response envelope
```json
{
  "success": true,
  "data": { },
  "error": null,
  "meta": { "request_id": "...", "next_cursor": null }
}
```
Error shape (`success:false`, `data:null`):
```json
{
  "success": false,
  "data": null,
  "error": { "code": "entitlement_exceeded", "message": "Free plan allows 1 portfolio.",
             "details": { "limit": 1 } },
  "meta": { "request_id": "..." }
}
```

### Error codes (canonical)
`validation_error` (422), `unauthenticated` (401), `forbidden` (403), `not_found` (404),
`entitlement_exceeded` (402/403), `rate_limited` (429), `conflict` (409), `infeasible` (422, optimizer),
`upstream_unavailable` (503, data vendor), `internal_error` (500).

### Compliance header
Research-output responses (scores, signals, optimizations, backtests) include
`X-QuantVista-Disclaimer: research-only; not investment advice` and a `disclaimer` field. (See `07`.)

## 3. Key endpoint contracts (representative)

### 3.1 Auth
- `POST /auth/register` → `{ email, password, name }` → tenant + owner user created (Free plan).
- `POST /auth/login` → tokens. `POST /auth/refresh`. `POST /auth/logout`.
- `GET /me` → user + active tenant + entitlements summary.

### 3.2 Stocks (reference)
`GET /stocks?market=NSE&sector=IT&filter[market_cap_bucket]=large&limit=50&cursor=...`
```json
{ "success": true, "data": [
  { "symbol": "RELIANCE", "company_name": "Reliance Industries",
    "sector": "Energy", "composite_score": 88.2, "as_of": "2026-06-13" }
], "meta": { "next_cursor": "..." } }
```
`GET /stocks/{symbol}` → master + latest snapshot (price, key fundamentals, latest scores, top news).

### 3.3 Scores & explainability
`GET /scores/{symbol}?as_of=2026-06-13`
```json
{ "success": true, "data": {
  "symbol": "RELIANCE", "as_of": "2026-06-13",
  "fundamental": 90, "momentum": 80, "quality": 85, "sentiment": 75, "risk": 70,
  "composite": 84.0, "weights_version": "v1", "model_version": "score-2026.06",
  "disclaimer": "Research signal, not investment advice." } }
```
`GET /scores/{symbol}/decomposition?as_of=2026-06-13` → per-factor contributions that **sum to** the
composite, each with `raw_value`, `zscore`, `percentile`, and the input's PIT `as_of` date (US-02 AC).

`GET /rankings?universe=NIFTY200&as_of=2026-06-13&sort=-composite&limit=50` → ranked list (Free tier:
truncated to top-50 per entitlements).

### 3.4 Screener
`POST /screener`
```json
{ "universe": "NIFTY200",
  "filters": [ {"field":"roe","op":"gte","value":15},
               {"field":"composite_score","op":"gte","value":70} ],
  "sort": "-momentum_score", "limit": 100 }
```
→ rows + `meta.count`. Saving: `POST /screens { name, criteria }` (entitlement-limited).

### 3.5 Portfolios & optimization
`POST /portfolios` (Idempotency-Key) → `{ name, benchmark, base_currency }`. Server enforces
`max_portfolios` entitlement → `entitlement_exceeded` (US-06 AC).

`POST /portfolios/{id}/optimize`
```json
{ "method": "mean_variance",
  "objective": "max_sharpe",
  "constraints": { "max_weight": 0.15, "sector_caps": {"IT": 0.25}, "target_volatility": 0.18,
                   "long_only": true },
  "candidate_universe": "current_positions" }
```
Success → weights summing to 1.0 + expected return/vol + per-constraint status. Infeasible →
`error.code = "infeasible"` with the binding constraint (US-03 AC). `method` ∈
`mean_variance | risk_parity | black_litterman | hrp` (gated by tier/phase).

`GET /portfolios/{id}/risk` → beta, vol, max drawdown, Sharpe, Sortino, sector exposure, concentration (HHI).
`POST /portfolios/{id}/rebalance` → suggested trades to reach target weights with drift threshold.

### 3.6 Backtesting (async)
`POST /backtests` (Quant tier for full range)
```json
{ "spec": { "type": "factor_strategy",
            "universe": "NIFTY200",
            "rules": { "rank_by": "composite", "top_n": 20, "rebalance": "monthly" },
            "start": "2018-01-01", "end": "2025-12-31",
            "costs_bps": 15, "benchmark": "NIFTY200_TRI" } }
```
→ `202 Accepted` with `{ "id": "...", "status": "queued" }`. Poll `GET /backtests/{id}` →
`status ∈ queued|running|succeeded|failed`; on success returns metrics (CAGR, vol, Sharpe, max DD, hit rate,
turnover) + a result artifact link. Methodology page documents PIT/survivorship controls (US-04 AC).

### 3.7 Alerts
`POST /alerts`
```json
{ "scope": "portfolio", "target_id": "<portfolio_id>",
  "condition": { "metric": "composite_score", "op": "lt", "value": 50 },
  "channel": "email" }
```
Alerts evaluated each scoring cycle; events deduplicated; tier limits enforced (US-05 AC).

### 3.8 Entitlements
`GET /entitlements` → resolved limits/flags for the current tenant's plan, e.g.
`{ "max_portfolios": 1, "backtest": false, "api_access": false, "universe_scores": "top_50" }`.

### 3.9 Billing
`POST /billing/checkout-session` → Stripe Checkout URL. `POST /billing/webhook` (Stripe → server) updates
`subscriptions` and invalidates the entitlement cache. `GET /billing/portal` → Stripe customer portal URL.

## 4. Versioning & deprecation

- URI version `/api/v1`. Breaking changes → `/api/v2` with an overlap window; deprecations announced via
  `Deprecation`/`Sunset` headers and changelog.
- Additive changes (new fields/endpoints) are non-breaking and ship within `v1`.

## 5. Public API (Quant tier)

- API-key auth (scoped, revocable), stricter rate limits, usage metering for billing.
- Read-only subset first: `/stocks`, `/scores`, `/rankings`, `/screener`. Documented OpenAPI + examples.
